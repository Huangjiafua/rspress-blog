# 密码哈希、bcrypt 与身份认证安全基础

本文从工程安全视角系统梳理密码哈希的核心问题、bcrypt 的设计原理，以及 Argon2、scrypt、PBKDF2 等相近方案的实现思想和选型边界。

## 1. 问题背景

密码存储的目标不是“加密后再解密”，而是让系统可以验证用户是否知道正确密码，同时即使数据库泄露，攻击者也很难批量还原原始密码。

因此，密码不应以明文保存，也不应直接使用普通哈希算法保存：

```text
MD5(password)
SHA1(password)
SHA256(password)
SHA256(password + salt)
```

普通哈希算法的设计目标是快速、稳定、低成本地生成摘要。这个特性适合文件校验、签名摘要和数据完整性验证，但不适合密码存储。数据库泄露后，攻击者可以使用 GPU、ASIC 或大规模并行计算设备高速尝试常见密码、字典组合和规则变形。

密码哈希算法的核心目标正好相反：它要让每一次密码猜测都变得昂贵。

一个合格的密码哈希方案通常需要具备：

| 能力 | 作用 |
| --- | --- |
| 随机盐 salt | 让相同密码产生不同哈希，抵抗彩虹表和批量预计算 |
| 慢计算 | 提高单次猜测成本，削弱暴力破解效率 |
| 成本参数 | 允许随硬件性能提升逐步提高计算成本 |
| 内存成本 | 增加 GPU、ASIC 大规模并行破解的硬件代价 |

## 2. bcrypt 是什么

bcrypt 是一种专门用于密码存储的自适应密码哈希算法。它基于 Blowfish 加密算法中的密钥调度过程构造，通过重复执行昂贵的密钥扩展来放大计算成本。

需要明确区分：

| 概念 | 是否可逆 | 典型用途 |
| --- | --- | --- |
| 加密 | 可逆，需要密钥 | 保护可恢复数据，例如配置密文、文件加密 |
| 普通哈希 | 不可逆，速度快 | 文件校验、签名摘要、数据完整性 |
| 密码哈希 | 不可逆，故意慢 | 存储和验证用户密码 |

bcrypt 不是加密算法，不能从 bcrypt 哈希中解密出原始密码。它的工程价值在于：当攻击者拿到密码哈希后，每一次离线猜测都必须付出较高计算成本。

## 3. bcrypt 哈希格式

一个典型 bcrypt 哈希如下：

```text
$2b$12$abcdefghijklmnopqrstuuK7YQFS3x5uA9K6BvGfC8LxYv0Yh8x6G
```

它可以拆成：

```text
$2b$12$salt.................hash......................
```

各部分含义：

| 字段 | 含义 |
| --- | --- |
| `$2b$` | bcrypt 版本标识 |
| `12` | cost 成本因子 |
| salt | 随机盐，通常直接嵌入哈希字符串 |
| hash | bcrypt 计算得到的密码哈希结果 |

salt 不需要保密。它的作用不是作为密钥，而是保证相同密码在不同用户、不同时间生成不同哈希。

例如两个用户都使用同一个密码：

```text
bcrypt("123456", saltA) -> hashA
bcrypt("123456", saltB) -> hashB
```

即使原始密码相同，最终哈希也不同。攻击者无法用一张通用预计算表同时匹配所有用户。

## 4. cost 成本因子

bcrypt 的 cost 控制计算成本。`cost=12` 并不表示循环 12 次，而是约等于执行 `2^12` 级别的昂贵密钥扩展。

```text
cost = 10 -> 2^10
cost = 12 -> 2^12
cost = 14 -> 2^14
```

cost 每增加 1，计算成本大约翻倍。它带来的效果是双向的：

| 影响对象 | 结果 |
| --- | --- |
| 正常用户登录 | 单次验证略变慢 |
| 攻击者离线破解 | 每一次猜测都变慢 |
| 应用服务器 | CPU 压力上升 |

工程上应通过压测选择 cost。常见策略是让单次哈希耗时保持在业务可接受范围内，例如几十到几百毫秒。不能盲目设置过高，否则可能造成登录接口延迟过大，甚至让认证服务更容易被资源消耗型攻击拖垮。

## 5. bcrypt 的实现原理

bcrypt 的核心来自 Blowfish 的密钥调度机制。Blowfish 本身是对称加密算法，内部有 P-array 和 S-box 等状态结构。bcrypt 利用的是 Blowfish 初始化和扩展密钥过程较慢这一特点，而不是直接把 Blowfish 当作普通加密工具使用。

简化流程如下：

```text
输入：password, salt, cost

1. 初始化 Blowfish 状态
2. 使用 password 和 salt 执行密钥扩展
3. 重复 2^cost 轮昂贵扩展
4. 使用最终状态加密固定文本
5. 输出 version、cost、salt、hash
```

更接近实现思想的伪流程：

```text
state = InitBlowfishState()

state = ExpandKey(state, salt, password)

repeat 2^cost times:
    state = ExpandKey(state, 0, password)
    state = ExpandKey(state, 0, salt)

hash = EncryptFixedText(state, "OrpheanBeholderScryDoubt")
```

其中 `ExpandKey` 是成本来源。它不断把 password 和 salt 混入 Blowfish 的内部状态，使攻击者无法用一次快速哈希完成猜测。

bcrypt 的系统设计要点是：验证密码时，系统从已保存的 bcrypt 字符串中解析版本、cost 和 salt，再用用户输入的密码重新计算一次 bcrypt。如果新结果与保存结果一致，则说明密码正确。

注册流程：

```text
password = 用户输入
salt = 随机生成
hash = bcrypt(password, salt, cost)
保存 hash
```

登录流程：

```text
password = 用户输入
stored_hash = 数据库中的 bcrypt 字符串

从 stored_hash 中解析 version、cost、salt
使用相同参数重新计算 bcrypt(password)
使用常量时间比较结果
```

## 6. bcrypt 的优势与限制

bcrypt 的优势：

| 优势 | 说明 |
| --- | --- |
| 面向密码存储设计 | 比普通哈希更适合身份认证场景 |
| 自动携带 salt | 常见实现会把 salt、cost、版本一起编码进结果 |
| 成本可调 | 可以随硬件性能提升逐步提高 cost |
| 实战验证充分 | 历史久，语言生态成熟，框架支持广 |

bcrypt 的限制：

| 限制 | 影响 |
| --- | --- |
| 通常只处理前 72 字节密码 | 超长密码可能被截断，需要了解具体库行为 |
| 主要消耗 CPU | 对现代 GPU/ASIC 的内存约束不如 Argon2、scrypt |
| 参数维度较少 | 只能主要调节 cost，不能直接调节内存成本 |
| 不适合通用摘要 | 不应用于文件校验、签名摘要或数据指纹 |

72 字节限制是实际工程中最容易忽略的问题。如果业务允许很长密码或 passphrase，应确认所用 bcrypt 库的处理方式。某些系统会先进行预哈希再 bcrypt，但这会引入新的编码、兼容性和协议设计问题，不能随意拼接实现。

## 7. Argon2

Argon2 是 Password Hashing Competition 的获胜算法，是现代密码哈希方案中非常重要的选择。它的核心设计目标是同时消耗计算资源和内存资源，从而提高 GPU、FPGA、ASIC 大规模破解的成本。

Argon2 有三个常见变体：

| 变体 | 特点 |
| --- | --- |
| Argon2d | 更强调抵抗 GPU 破解，但可能暴露数据相关内存访问特征 |
| Argon2i | 更强调抗侧信道攻击，内存访问模式更独立 |
| Argon2id | 混合模式，通常作为通用密码存储首选 |

Argon2id 的参数通常包括：

| 参数 | 含义 |
| --- | --- |
| memory_cost | 使用多少内存 |
| time_cost | 迭代轮数 |
| parallelism | 并行度 |

简化流程：

```text
输入 password, salt, memory_cost, time_cost, parallelism

1. 初始化大块内存区域
2. 根据 password 和 salt 填充内存块
3. 多轮访问、混合、压缩内存块
4. 输出最终哈希
```

Argon2id 的优势是安全模型更贴近现代硬件攻击现实。攻击者不仅需要算力，还需要大量高速内存。对于新系统，如果运行环境和语言生态支持良好，Argon2id 通常是优先选择。

## 8. scrypt

scrypt 也是 memory-hard 的密码哈希和密钥派生算法。它通过生成并随机访问大量内存块，迫使攻击者在大规模并行破解时投入更多内存资源。

scrypt 的核心参数包括：

| 参数 | 含义 |
| --- | --- |
| N | CPU 和内存成本，通常是 2 的幂 |
| r | 块大小参数 |
| p | 并行度参数 |

简化流程：

```text
1. 使用 PBKDF2 做初始派生
2. 生成大量伪随机内存块
3. 反复随机访问和混合这些内存块
4. 再使用 PBKDF2 输出最终结果
```

scrypt 的优势是比 bcrypt 更强调内存成本，适合需要抵抗专用硬件批量破解的场景。它的缺点是参数配置比 bcrypt 更复杂，错误配置可能导致安全性不足或资源消耗过高。

## 9. PBKDF2

PBKDF2 是经典的密码派生函数，常用于从用户密码派生加密密钥，也可以用于密码存储。它通常基于 HMAC-SHA256 或 HMAC-SHA512 反复迭代。

简化流程：

```text
输入 password, salt, iterations

repeat iterations times:
    使用 HMAC 反复计算和混合

输出派生密钥
```

PBKDF2 的优势是标准化早、兼容性强、平台支持广。许多协议、合规环境和老系统都内置支持 PBKDF2。

它的主要限制是缺少强内存成本约束。PBKDF2 主要依赖大量 CPU 迭代，面对 GPU、ASIC 这类并行硬件时，同等资源下的防护能力通常不如 Argon2id 或 scrypt。

## 10. 选型建议

新系统的推荐顺序通常可以按以下思路理解：

```text
Argon2id > scrypt > bcrypt > PBKDF2 > 普通哈希
```

更实际的工程选型：

| 场景 | 建议 |
| --- | --- |
| 新系统，运行环境支持良好 | 优先使用 Argon2id |
| 通用 Web 系统，生态中 bcrypt 支持成熟 | bcrypt 仍然可靠 |
| 需要 memory-hard，但没有 Argon2 支持 | 考虑 scrypt |
| 受限于合规、协议或平台能力 | 使用足够迭代次数的 PBKDF2 |
| 只想快速摘要密码 | 不应使用 MD5、SHA1、单次 SHA256 |

密码存储不应自行发明算法，也不应手写密码哈希流程。应优先使用成熟框架或可信密码学库，并明确检查默认参数是否符合当前安全要求。

## 11. 常见误区

### 误区一：加 salt 的 SHA256 就足够安全

salt 解决的是相同密码同哈希和彩虹表预计算问题，但不能解决普通哈希太快的问题。

```text
SHA256(password + salt)
```

仍然可以被攻击者高速批量尝试。密码哈希必须同时具备 salt 和成本控制。

### 误区二：bcrypt 越慢越安全，所以 cost 越高越好

cost 需要结合服务器资源、登录频率、并发量和攻击面压测。过高的 cost 可能让正常认证服务变慢，也可能放大拒绝服务风险。

### 误区三：hash 结果需要加密保存

密码哈希本身就是用于不可逆验证的结果。通常不需要再加密保存 bcrypt 字符串。更重要的是保护数据库、控制访问权限、使用 TLS、做好密钥和日志治理。

### 误区四：密码哈希可以替代访问令牌保护

密码哈希用于保存用户密码，不等于可以安全保存 API Token、Session Token 或 OAuth Refresh Token。不同秘密材料有不同生命周期和验证方式，需要分别设计。

## 12. 总结

bcrypt 的本质是：使用随机 salt 和可调 cost，让每一次密码猜测都变得昂贵。它通过 Blowfish 昂贵的密钥调度过程实现慢哈希，是经过长期实战验证的密码存储方案。

但从现代安全工程角度看，攻击者已经可以使用大规模并行硬件。新系统如果条件允许，应优先考虑 Argon2id 这类同时约束 CPU 和内存成本的方案。bcrypt 仍然是成熟可靠的选择，但需要理解它的 72 字节限制、CPU 型成本模型和参数调优边界。

最终原则是：

```text
不要明文保存密码
不要用普通哈希保存密码
不要自创密码哈希算法
使用成熟库
为每个密码使用随机 salt
选择可调成本
定期评估参数是否需要升级
```

## 13. 对比表

| 算法 | 类型 | 核心防护 | 关键参数 | 适用性 |
| --- | --- | --- | --- | --- |
| bcrypt | 密码哈希 | CPU 慢计算 | cost | 成熟可靠，通用 Web 场景常见 |
| Argon2id | 密码哈希 / KDF | CPU + 内存困难 | memory、time、parallelism | 新系统优先选择 |
| scrypt | KDF / 密码哈希 | 内存困难 | N、r、p | 抗硬件破解能力较好，参数较复杂 |
| PBKDF2 | KDF | 大量 HMAC 迭代 | iterations | 兼容性强，安全性依赖高迭代 |
| SHA256 | 普通哈希 | 快速摘要 | 无密码成本 | 不适合直接保存密码 |
| MD5 / SHA1 | 普通哈希 | 快速摘要 | 无密码成本 | 不应再用于密码安全场景 |
